# 依赖注入规范文档

## 概述

本文档描述 MathFun 后端项目采用的依赖注入方案，基于 Google Wire 框架实现。

## 技术选型

- **框架**：Google Wire
- **版本**：v0.7.0
- **官网**：https://github.com/google/wire

## 目录结构

```
backend/internal/infrastructure/wire/
├── providers.go       # 基础 Provider（Config, DB, JWT, EventBus 等）
├── user.go          # User 模块 Wire 定义（wireinject）
├── knowledge.go     # Knowledge 模块 Wire 定义（wireinject）
└── wire_gen.go     # 自动生成的依赖注入代码
```

### 公共包的注册

`pkg/` 目录下的公共包（如 errors、response、validator、query、metrics）不需要在 Wire 中注册，
因为它们是通过函数参数直接传递的，不属于依赖注入范畴：

```go
// 这些是工具包，通过函数参数传递，不需要 Wire 注册
func InitializeUserModule(
    db *gorm.DB,
    logger *zap.Logger,
    // errors、response 等通过参数传入
) (*service.Service, error) {
    // ...
}
```

## 模块划分原则

采用**功能模块划分**的混合模式：

- **公共基础层**：providers.go - 跨模块共享的基础设施
- **功能模块层**：user.go, knowledge.go 等 - 按业务功能划分

### 功能模块划分 vs 层级划分

| 划分方式 | 改动影响范围 | 适用场景 |
|----------|-------------|---------|
| 功能模块 | 仅影响当前模块 | 业务驱动的微服务 |
| 层级划分 | 影响整个依赖链 | 底层库或工具 |

**选择理由**：功能模块划分使改动范围可控，构建更快。

## 使用规范

### 1. 创建新模块

在 `wire/` 目录下创建新模块文件，如 `invitation.go`：

```go
//go:build wireinject
// +build wireinject

package wire

import (
	"github.com/google/wire"
	"gorm.io/gorm"
	"go.uber.org/zap"

	"mathfun/internal/application/invitation/service"
	"mathfun/internal/infrastructure/persistence/gorm/repo"
)

// InitializeInvitationModule 初始化 Invitation 模块
func InitializeInvitationModule(
	db *gorm.DB,
	logger *zap.Logger,
) (*service.Service, error) {
	wire.Build(
		repo.NewInvitationRepository,
		service.NewService,
	)
	return nil, nil
}
```

### 2. 运行 Wire 生成

```bash
cd backend/internal/infrastructure/wire
go run github.com/google/wire/cmd/wire@latest ./
```

生成的代码会自动写入 `wire_gen.go`。

### 3. 在 Server 中使用

```go
// server/service.go
func (s *ServerService) registerRoutes() {
	// 初始化模块
	invitationSvc, err := wire.InitializeInvitationModule(s.db, s.logger)
	if err != nil {
		s.logger.Error("初始化 Invitation 模块失败", zap.Error(err))
		return
	}
	
	// 创建 Handler 并注册路由
	handler := invitation.NewHandler(invitationSvc, s.logger)
	invitation.RegisterRoutes(api, handler)
}
```

## 关键约束

### Build Tag 要求

- Wire 定义文件必须使用 `//go:build wireinject` 标签
- 生成的 `wire_gen.go` 必须使用 `//go:build !wireinject` 标签
- 这样可以避免 Wire 工具在编译时参与

### 跨包类型问题

当 Wire 需要返回跨包的接口/结构体时，可能遇到类型解析问题。

**解决方案**：
- 将 Handler 初始化逻辑放在 wire 包中
- 只返回 Service 层，而不是 Handler 层
- Handler 在 server/service.go 中手动创建

### 事件总线适配

不同模块的 EventBus 接口可能不兼容，需要适配器：

```go
// user.go
type userEventBusAdapter struct {
	bus *event.EventBus
}

func newUserEventBusAdapter(bus *event.EventBus) appService.EventBus {
	return &userEventBusAdapter{bus: bus}
}
```

## 版本控制

`wire_gen.go` 必须提交到版本库：

- 避免每次 clone 后都要运行 `go generate`
- 生成的代码是确定性的
- CI 构建时可以跳过 wire 步骤

## 扩展场景

### 添加新依赖

1. 在对应的 wire 模块文件中添加 Provider
2. 运行 `go run github.com/google/wire/cmd/wire@latest ./...`
3. 更新使用方代码

### 修改依赖关系

1. 修改 wire 模块文件中的 `wire.Build()` 调用
2. 运行 Wire 重新生成
3. 检查生成的代码是否符合预期

## 常见问题

### Q: Wire 报错 "no provider found"

A: 检查以下几点：
- 依赖的构造函数是否在 wire.Build() 中声明
- 返回类型是否正确（指针 vs 值）
- 是否存在循环依赖

### Q: 如何调试 Wire 生成结果

A: 查看 `wire_gen.go` 文件，确认生成的代码逻辑是否正确。

### Q: 可以不使用 Wire 吗？

A: 可以。Wire 是可选的，也可以直接手动创建依赖。providers.go 提供了标准化的初始化函数，可以单独使用。
