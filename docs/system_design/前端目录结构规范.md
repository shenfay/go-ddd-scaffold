# MathFun 前端目录结构规范 (5 层架构版)

## 1. 概述

本文档定义了 MathFun 前端项目的核心架构标准：**5 层架构体系**。该架构旨在通过严格的职责划分，实现 UI 渲染、交互逻辑、业务规则、数据获取和基础能力之间的解耦，从而提升代码的可维护性、可测试性和多端适配能力。

## 2. 核心 5 层架构定义

### 2.1 表现层 (Presentation Layer)
- **职责**: 负责纯 UI 渲染、页面路由管理、主题样式切换。
- **核心原则**: 不包含业务逻辑，仅处理“如何显示”。
- **目录**: `src/presentation/`
  - `pages/`: 页面级组件。
  - `components/`: 纯 UI 基础组件（如 Button, Input, Modal, Layout）。
  - `routes/`: 路由配置。
  - `styles/`: 全局 CSS、主题变量定义。

### 2.2 交互层 (Interaction Layer)
- **职责**: 处理用户交互反馈、3D 场景控制、手势/点击事件监听。
- **核心原则**: 作为表现层与业务逻辑层的桥梁，处理“交互反馈”。
- **目录**: `src/interaction/`
  - `three/`: Three.js 场景、模型加载、动画控制逻辑。
  - `components/`: 带有复杂交互逻辑的容器组件（如 NPC 对话框、学习卡片）。
  - `hooks/`: 专用交互 Hooks（如 `useClickInteraction`, `useDragDrop`）。

### 2.3 业务逻辑层 (Business Layer)
- **职责**: 定义业务规则、管理全局状态、处理核心业务流程（如学习闭环）。
- **核心原则**: 独立于具体框架和技术实现，处理“怎么运行”。
- **目录**: `src/business/`
  - `store/`: 全局状态管理（Slices, Selectors, Middleware）。
  - `models/`: 领域模型定义（C/S/T/P 节点数据结构）。
  - `services/`: 核心业务逻辑服务（如 `LearningFlowService`）。

### 2.4 数据接入层 (Data Access Layer)
- **职责**: 处理 API 请求、数据解析与序列化、缓存策略、错误拦截。
- **核心原则**: 封装所有外部数据来源，处理“数据来源”。
- **目录**: `src/data/`
  - `api/`: API 客户端配置（Axios 实例、拦截器）。
  - `endpoints/`: 接口路径常量定义。
  - `repositories/`: 数据仓储，提供具体的数据获取方法。

### 2.5 基础能力层 (Base/Shared Layer)
- **职责**: 提供跨层级复用的通用工具、底层协议、日志管理、安全加固。
- **核心原则**: 处理“底层支撑”，不依赖于上层任何业务。
- **目录**: `src/shared/`
  - `utils/`: 通用工具（SoundManager, Logger, TextureLoader）。
  - `types/`: 全局 TypeScript 类型定义。
  - `constants/`: 系统级常量（非业务相关）。
  - `hooks/`: 通用底层 Hooks（如 `useLocalStorage`, `useDebounce`）。

## 3. 标准目录树

```
frontend/
├── public/
├── src/
│   ├── presentation/       # 1. 表现层
│   │   ├── pages/
│   │   ├── components/
│   │   ├── routes/
│   │   └── styles/
│   ├── interaction/        # 2. 交互层
│   │   ├── components/
│   │   ├── three/
│   │   └── hooks/
│   ├── business/           # 3. 业务逻辑层
│   │   ├── store/
│   │   ├── models/
│   │   └── services/
│   ├── data/               # 4. 数据接入层
│   │   ├── api/
│   │   ├── repositories/
│   │   └── endpoints/
│   ├── shared/             # 5. 基础能力层
│   │   ├── utils/
│   │   ├── types/

├── package.json
└── tailwind.config.js
```

## 4. 迁移与开发规范

1. **单向依赖**: 依赖关系应从上至下（表现层 -> 交互层 -> 业务层 -> 数据层 -> 基础层），禁止循环依赖。
2. **命名规范**:
   - 目录与组件：PascalCase (e.g., `NPCInteraction/`)
   - 工具与逻辑：camelCase (e.g., `soundManager.js`)
3. **清理要求**: 在重构过程中，若发现与 5 层架构不符的旧目录（如原 `src/features`），应将其内容拆分迁移至对应层级并删除原目录。

## 5. 总结

5 层架构是 MathFun 前端开发的“宪法”。所有新功能的开发和旧代码的维护都必须严格遵守此分层标准，以确保项目在快速迭代中保持架构的可控性和扩展性。
