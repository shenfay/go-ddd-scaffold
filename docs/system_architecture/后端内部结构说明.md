# Backend Internal 目录结构及文件作用说明

## 📁 整体目录结构

```
backend/internal/
├── application/          # 应用层
├── domain/              # 领域层
├── infrastructure/      # 基础设施层
├── interfaces/          # 接口层
└── pkg/                 # 共享包
```

## 🎯 各层文件详细说明

### 1. 🏗️ Application Layer (应用层)

#### `/application/knowledge/`
- **service/** - 知识图谱应用服务
  - `service.go` - 应用服务实现
    - 作用：协调领域服务，处理用例逻辑
    - 上下文：业务用例的入口点，编排领域对象
    - 交互：调用领域服务、仓储接口

- **dto/** - 数据传输对象
  - `dto.go` - DTO定义和转换函数
    - 作用：定义应用层与接口层的数据传输对象
    - 上下文：API响应格式、请求参数封装
    - 交互：领域实体与DTO之间的转换

#### `/application/user/`
- **service/** - 用户应用服务
  - `service.go` - 应用服务实现
    - 作用：处理用户相关的业务用例
    - 上下文：用户注册、登录、权限管理等
    - 交互：调用用户领域服务、JWT服务

- **dto/** - 数据传输对象
  - `dto.go` - DTO定义和转换函数
    - 作用：定义用户相关的请求/响应DTO
    - 上下文：用户API的数据传输
    - 交互：用户实体与DTO之间的转换

### 2. 🏭 Domain Layer (领域层)

#### `/domain/knowledge/`

##### `aggregate/`
- **domain_aggregate.go** - 领域聚合根
  - 作用：管理Domain及其Trunk集合，维护聚合内一致性
  - 上下文：知识领域业务规则的核心
  - 交互：包含领域创建、主干管理等业务逻辑

- **node_aggregate.go** - 节点聚合根
  - 作用：管理Node及其关系，维护节点间一致性
  - 上下文：知识点间的关联关系管理
  - 交互：处理节点关系建立、前置关系验证等

- **domain_aggregate_test.go** - 领域聚合根测试
  - 作用：验证领域聚合根的业务规则
  - 上下文：确保业务逻辑正确性

- **node_aggregate_test.go** - 节点聚合根测试
  - 作用：验证节点聚合根的业务规则
  - 上下文：确保节点关系逻辑正确

##### `entity/`
- **core_entities.go** - 核心实体定义
  - 作用：定义Domain、Trunk、Node等核心实体
  - 上下文：知识图谱的基础数据结构
  - 交互：被聚合根包含和管理

- **learning_entities.go** - 学习相关实体

  - 上下文：学生学习行为的数据模型

- **relationship_entities.go** - 关系实体

  - 上下文：知识点依赖关系的表示

##### `event/`
- **kg_events.go** - 知识图谱领域事件

##### `repository/`
- `repository.go` - 仓储接口定义

##### `service/`
- **kg_domain_service.go** - 知识图谱领域服务

  - 上下文：学习路径验证、关系建立等
  - 交互：协调多个聚合根的操作

- **kg_domain_service_test.go** - 领域服务测试
  - 作用：验证领域服务的业务逻辑

##### `valueobject/`

## 🔗 层间交互关系

### 数据流向
```
HTTP请求 → HTTP Handler → Application Service → Domain Service/Aggregate → Repository → Database
    ↑                                                                           ↓
    ←───────── HTTP响应 ←─────── 应用服务 ←──────── 领域逻辑 ←──────── 仓储实现 ←──────
```

### 依赖关系
- **HTTP Handler** 依赖 **Application Service**
- **Application Service** 依赖 **Domain Service** 和 **Repository接口**
- **Domain Service** 依赖 **Aggregate/Entity**
- **Repository实现** 依赖 **Database Models**
- **Models** 通过转换函数与 **Entities** 交互

## 🎯 设计原则体现

1. **依赖倒置**：高层模块不依赖低层模块
2. **单一职责**：每层每文件职责明确
3. **开闭原则**：对扩展开放，对修改封闭
4. **接口隔离**：通过接口隔离具体实现
5. **领域驱动**：业务逻辑集中在领域层
