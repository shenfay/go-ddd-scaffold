## 📡 详细请求处理流程

```mermaid
sequenceDiagram
    participant Client as 📱 客户端
    participant Gateway as 🌐 API网关
    participant Auth as 🔐 认证服务
    participant Router as 🔄 路由器
    participant Handler as 🎯 Handler
    participant Service as 💼 应用服务
    participant Domain as 🏭 领域服务
    participant Aggregate as 📦 聚合根
    participant Repo as 💾 仓储
    participant DB as 🗄️ 数据库

    Client->>Gateway: 📤 HTTP请求
    Gateway->>Auth: 🔍 JWT Token验证
    Auth-->>Gateway: ✅ 认证结果
    Gateway->>Router: 📍 路径匹配
    Router->>Handler: 🎯 调用处理器
    Handler->>Service: 📞 调用应用服务
    Service->>Domain: 🔧 调用领域服务
    Domain->>Aggregate: 📦 操作聚合根
    Aggregate->>Repo: 💾 数据持久化
    Repo->>DB: 🗄️ 数据库操作
    DB-->>Repo: 📤 查询结果
    Repo-->>Aggregate: 📥 聚合数据
    Aggregate-->>Domain: 🔄 业务逻辑处理
    Domain-->>Service: ✅ 处理结果
    Service-->>Handler: 📊 服务响应
    Handler-->>Router: 📍 响应包装
    Router-->>Gateway: 🔁 格式化响应
    Gateway-->>Client: 📥 HTTP响应

    note over Client,DB: 📊 完整请求响应周期
```

## 🔧 API请求处理链路

```mermaid
flowchart TD
    A[📥 HTTP请求到达] --> B{🔍 是否需要认证?}
    B -->|是| C[🔐 JWT Token解析]
    B -->|否| D[🔄 路径路由]
    C --> E{✅ Token有效?}
    E -->|是| F[👤 用户信息提取]
    E -->|否| G[❌ 401 Unauthorized]
    F --> D
    D --> H[🎯 路由匹配]
    H --> I[🧩 参数绑定]
    I --> J[🧪 请求验证]
    J --> K{✅ 验证通过?}
    K -->|是| L[🎯 调用Handler]
    K -->|否| M[❌ 400 Bad Request]
    L --> N[💼 应用服务调用]
    N --> O[🏭 领域逻辑处理]
    O --> P[💾 数据持久化]
    P --> Q[📤 响应构建]
    Q --> R[📦 数据序列化]
    R --> S[📤 HTTP响应返回]

    style A fill:#4CAF50
    style G fill:#F44336
    style M fill:#F44336
    style S fill:#2196F3
```

## 🎯 流程优化建议

### 当前优势
- ✅ **分层清晰**: 严格遵循Clean Architecture
- ✅ **解耦良好**: 各层职责分明
- ✅ **可测试性强**: 每层都易于单元测试
- ✅ **扩展性好**: 易于添加新功能

### 优化方向
1. **缓存策略**: 添加Redis缓存热点数据
2. **异步处理**: 耗时操作异步化
3. **监控完善**: 添加APM性能监控
4. **日志优化**: 结构化日志便于排查
