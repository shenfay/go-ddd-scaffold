## ⚠️ 识别的不规范之处

### 高优先级问题 🔴

#### 1. 目录结构不统一
**问题：** `internal/app/` vs `internal/application/`  
**影响：** 命名不一致，容易混淆  
**优化方案：** 统一使用 `application/`  
**工作量：** 1 小时  
**风险：** 低

#### 2. 测试覆盖率不足
**问题：** Domain 层 45%，Application 层 38%，Infrastructure 层 25%  
**目标：** Domain ≥ 90%, Application ≥ 80%, Infrastructure ≥ 60%  
**优化方案：** 分 3 周补充单元测试  
**工作量：** 40 小时  
**风险：** 低

### 中优先级问题 🟡

#### 3. Repository 位置混乱
**问题：** Repository 接口在 Domain 和 Application 层都有定义  
**影响：** 职责不清，违反 DDD 原则  
**优化方案：** 统一在 Domain 层定义（推荐）或 Application 层定义  
**工作量：** 4 小时  
**风险：** 中

#### 4. Ports 包结构混乱
**问题：** 分类标准不一致（有的按功能，有的按技术）  
**影响：** 难以查找和维护  
**优化方案：** 统一按领域/功能分类  
**工作量：** 6 小时  
**风险：** 中

#### 5. 错误处理不统一
**问题：** 存在 4 种不同的错误处理方式  
**影响：** 错误堆栈不清晰，调试困难  
**优化方案：** 统一三层错误处理策略  
**工作量：** 6 小时  
**风险：** 中

#### 6. 事件机制不完善
**问题：** 缺少重试机制、Dead Letter Queue、事件处理器注册混乱  
**影响：** 事件丢失风险，系统可靠性低  
**优化方案：** 实现 Outbox Pattern、重试机制、自动注册  
**工作量：** 16 小时  
**风险：** 中

### 低优先级问题 🟢

#### 7. 命名不一致
**问题：** ID 生成器有的叫 Generator，有的叫 Node  
**影响：** API 不一致  
**优化方案：** 统一使用 Generator  
**工作量：** 2 小时  
**风险：** 低

#### 8. 注释不完整
**问题：** 部分导出的类型和函数缺少注释  
**影响：** IDE 提示不友好，godoc 文档不完整  
**优化方案：** 添加注释要求 + golint 检查  
**工作量：** 8 小时  
**风险：** 低

#### 9. 配置管理分散
**问题：** 配置分散在 config.yaml、.env 等多个文件  
**影响：** 配置来源不清晰，优先级混乱  
**优化方案：** 使用 Viper 统一管理，明确优先级  
**工作量：** 8 小时  
**风险：** 中

#### 10. Module 组装过于复杂
**问题：** NewAuthModule 等函数长达 50+ 行  
**影响：** 难以理解和维护，违反单一职责  
**优化方案：** 拆分组装逻辑为多个私有方法  
**工作量：** 6 小时  
**风险：** 低
